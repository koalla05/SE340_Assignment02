# SE340_Assignment02

## 1. Identified Performance Issues & Detection Methods
* **Issue 1: Substring Full Table Scans** * *Detection:* `EXPLAIN ANALYZE` showed a `Seq Scan` on `customers` taking 3.7ms due to `LIKE '%gmail%'`.
* **Issue 2: Large-Volume Joins**
    * *Detection:* High execution time (59.1ms) on product revenue join due to joining raw table rows before calculating aggregates.
* **Issue 3: Write Bloat on Wide Table**
    * *Detection:* `EXPLAIN ANALYZE` on `customer_events_wide` showed heavy `Heap Fetches` despite an active composite index, caused by background updates dirtying table pages.
* **Issue 4: Lock Contention (Bonus Task)**
    * *Detection:* Queries against `pg_stat_activity` and `pg_locks` revealed multiple worker PIDs waiting on `ExclusiveLock` status during concurrent updates.

## 2. Implemented Optimizations
* **Indexing Strategy:** Applied Trigram (`pg_trgm`) GIN indexes for substring lookups and built composite covering indexes (`idx_order_items_covering_perf`) to support Index-Only Scans.
* **Query Rewriting:** Transformed the product revenue aggregation query to execute an inline subquery pre-aggregation step, reducing join cardinality down to 2,000 distinct product keys.
* **Database Maintenance:** Applied manual `VACUUM ANALYZE` operations to clear dead rows generated by concurrent traffic and update the Visibility Map.

## 3. Structural & Architecture Recommendations
* **Table Splitting:** To permanently isolate the analytical event logs from high-frequency transactional updates, it is recommended to split `customer_events_wide` into a core event logging table and a peripheral attribute modification table.
